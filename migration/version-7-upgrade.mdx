---
title: "Migrate to Pulumi AWS Provider v7"
description: "Step-by-step guide to upgrading from Pulumi AWS Provider v6 to v7, covering breaking changes, removed resources, and enhanced region support."
---

Version 7.0.0 of the Pulumi AWS Provider is a major release that introduces enhanced region support, IAM role chaining, unified S3 resources, and several breaking changes aligned with the upstream Terraform AWS provider. This guide covers everything you need to know to migrate from v6.x to v7.0.0.

## Prerequisites

<Warning>
Before upgrading to v7.0.0, first upgrade to the latest available v6.x release of the provider. Skipping this step may cause unexpected behavior during migration.
</Warning>

Run `pulumi preview` on your latest v6.x code and confirm that:

- Your preview completes without errors or unexpected changes.
- There are no deprecation warnings related to the changes described in this guide.

It is also recommended that you run `pulumi up --refresh` before upgrading to ensure your Pulumi state is up to date with the actual AWS resources.

If you use version constraints in `package.json` or `requirements.txt`, update them to allow the `7.x` series and install the new version.

<CodeGroup>

```json package.json (before)
{
  "dependencies": {
    "@pulumi/aws": "^6.92.0"
  }
}
```

```json package.json (after)
{
  "dependencies": {
    "@pulumi/aws": "^7.0.0"
  }
}
```

</CodeGroup>

## Performing the upgrade

<Steps>
  <Step title="Update the provider version">
    Update your dependency version constraint to `^7.0.0` and install:

    <CodeGroup>

    ```bash npm
    npm install @pulumi/aws@^7.0.0
    ```

    ```bash pip
    pip install pulumi-aws --upgrade
    ```

    </CodeGroup>
  </Step>
  <Step title="Update your code for breaking changes">
    Review the breaking changes listed in this guide and update your Pulumi programs accordingly before running the upgrade. Pay special attention to provider role chaining, S3 bucket properties, and any resources you use.
  </Step>
  <Step title="Run pulumi up with --refresh and --run-program">
    After upgrading the library, run Pulumi with both the `--refresh` and `--run-program` flags:

    ```bash
    pulumi up --refresh --run-program
    ```

    Both flags are required:
    - `--refresh` migrates your state to the new schema so no spurious diffs are calculated.
    - `--run-program` ensures that the refresh uses the **new** provider version rather than the old one. Without this flag, `--refresh` would use the old provider.
  </Step>
  <Step title="Verify no unexpected diff">
    Confirm that the update shows no unexpected resource changes. You may see a refresh diff as Pulumi migrates state to the new structure, but you should not see an update diff for unchanged resources.
  </Step>
</Steps>

## Removed provider arguments

<Warning>
Remove the following endpoint overrides from your provider configuration. They are no longer supported and will cause errors if present.
</Warning>

- `endpoints.opsworks` — removed following AWS OpsWorks Stacks End of Life.
- `endpoints.simpledb` and `endpoints.sdb` — removed due to the removal of Amazon SimpleDB support.
- `endpoints.worklink` — removed due to the removal of Amazon Worklink support.

## Enhanced region support

Version 7.0.0 adds a `region` property to most resources, making it significantly easier to manage infrastructure across AWS regions without requiring multiple provider configurations. You can now set the region per resource rather than instantiating a separate `Provider` for each region.

<Note>
If you use the `aws.s3.Bucket` or `aws.s3.getBucket` resources, be aware that `region` now refers to the enhanced region support field. Use `bucketRegion` to get the region of the bucket itself.
</Note>

## Provider role chaining

v7 adds support for IAM role chaining by changing the `assumeRole` property to `assumeRoles` (a list), allowing multiple roles to be chained in sequence.

<Warning>
If you set `assumeRole` on a `Provider`, you must rename it to `assumeRoles` and wrap the value in an array. This is a breaking change.
</Warning>

<CodeGroup>

```typescript TypeScript (before v7)
import * as aws from '@pulumi/aws';

const provider = new aws.Provider("provider", {
    assumeRole: { roleArn: baseRole.arn },
});
```

```typescript TypeScript (v7+)
import * as aws from '@pulumi/aws';

const provider = new aws.Provider("provider", {
    assumeRoles: [{ roleArn: baseRole.arn }],
});
```

</CodeGroup>

If you use stack config instead of code, update `aws:assumeRole` to `aws:assumeRoles`:

<CodeGroup>

```yaml Pulumi.yaml (before v7)
config:
  aws:assumeRole:
    roleArn: arn:aws:iam::12345678912/someRole
```

```yaml Pulumi.yaml (v7+)
config:
  aws:assumeRoles:
    - roleArn: arn:aws:iam::12345678912/someRole
```

</CodeGroup>

## S3 Bucket/BucketV2 changes

In v6, `s3.Bucket` and `s3.BucketV2` represented different implementations. In v7, both resources now point to the same upstream implementation (the latest Terraform S3 Bucket resource). This unification introduces SDK-level breaking changes to `s3.Bucket`.

### Breaking changes to `s3.Bucket`

<Warning>
The following `s3.Bucket` property changes are breaking and require code updates before running the upgrade.
</Warning>

- `loggings` (array) has been renamed to `logging` (single object).
- `website.routingRules` now only accepts a `string` value (JSON-encoded), not an array of objects.

<CodeGroup>

```typescript TypeScript (before v7)
import * as aws from '@pulumi/aws';

const migrationBucket = new aws.s3.Bucket("migrationBucket", {
    loggings: [{
        targetBucket: loggingBucket.bucket,
        targetPrefix: "/log",
    }],
    website: {
        indexDocument: "index.html",
        errorDocument: "error.html",
        routingRules: [{
            Condition: { KeyPrefixEquals: "docs" },
            Redirect: { ReplaceKeyPrefixWith: "documents/" }
        }],
    },
});
```

```typescript TypeScript (v7+)
import * as aws from '@pulumi/aws';

const migrationBucket = new aws.s3.Bucket("migrationBucket", {
    logging: {
        targetBucket: loggingBucket.bucket,
        targetPrefix: "/log",
    },
    website: {
        indexDocument: "index.html",
        errorDocument: "error.html",
        routingRules: `[{
          "Condition": { "KeyPrefixEquals": "docs" },
          "Redirect": { "ReplaceKeyPrefixWith": "documents/" }
        }]`,
    },
});
```

</CodeGroup>

## Tags changes

In v6, Pulumi-level tag customizations caused the `tags` property to return _all_ tags on a resource, including provider-level default tags. In v7, this behavior has been removed to align with the upstream Terraform provider.

<Warning>
If your code reads `tags` to get all tags (including default tags), you must update it to use `tagsAll` instead.
</Warning>

<CodeGroup>

```typescript TypeScript (before v7)
const provider = new aws.Provider('provider', {
    defaultTags: { tags: { globalTag: 'value' } }
});
const bucket = new aws.s3.Bucket('my-bucket', {
    tags: { resourceTag: 'value' }
}, { provider });

// In v6, bucket.tags contained both globalTag AND resourceTag
export const tags = bucket.tags;
```

```typescript TypeScript (v7+)
const provider = new aws.Provider('provider', {
    defaultTags: { tags: { globalTag: 'value' } }
});
const bucket = new aws.s3.Bucket('my-bucket', {
    tags: { resourceTag: 'value' }
}, { provider });

// In v7, bucket.tags only contains resource-level tags
export const tags = bucket.tags;
// Use tagsAll to get all tags including defaults
export const allTags = bucket.tagsAll;
```

</CodeGroup>

## Key resource breaking changes

The sections below cover the most impactful per-resource changes. See the full list of all resource changes in the accordion section below.

### `aws.apigateway.Deployment`

<Warning>
The `stageName`, `stageDescription`, and `canarySettings` properties have been removed. You must use `aws.apigateway.Stage` explicitly.
</Warning>

The `invokeUrl` and `executionArn` outputs have also been removed from `Deployment`; read them from `aws.apigateway.Stage` instead.

<CodeGroup>

```typescript TypeScript (before v7)
const exampleDeployment = new aws.apigateway.Deployment("example", {
    restApi: exampleRestApi.id,
    stageName: "prod",
});

export const invokeUrl = exampleDeployment.invokeUrl;
export const executionArn = exampleDeployment.executionArn;
```

```typescript TypeScript (v7+)
const exampleDeployment = new aws.apigateway.Deployment("example", {
    restApi: exampleRestApi.id,
});

const prodStage = new aws.apigateway.Stage("prod", {
    stageName: "prod",
    restApi: exampleRestApi.id,
    deployment: exampleDeployment.id,
});

export const invokeUrl = prodStage.invokeUrl;
export const executionArn = prodStage.executionArn;
```

</CodeGroup>

If you previously relied on an implicitly created stage, import it before running `pulumi up`:

```bash
pulumi import aws:apigateway/stage:Stage prod restApi/stageName
```

### `aws.batch.JobQueue`

Remove `computeEnvironments` and replace it with `computeEnvironmentOrders`. Pulumi will migrate existing state automatically, but you must update your code.

<CodeGroup>

```typescript TypeScript (before v7)
const example = new aws.batch.JobQueue("example", {
    computeEnvironments: [exampleComputeEnvironment.arn],
    name: "patagonia",
    priority: 1,
    state: "ENABLED",
});
```

```typescript TypeScript (v7+)
const example = new aws.batch.JobQueue("example", {
    computeEnvironmentOrders: [{
        computeEnvironment: exampleComputeEnvironment.arn,
        order: 0,
    }],
    name: "patagonia",
    priority: 1,
    state: "ENABLED",
});
```

</CodeGroup>

### `aws.ec2.Instance`

<Warning>
`userData` is no longer hashed and is now stored in plaintext. Do not include passwords or sensitive information in `userData`.
</Warning>

- `userData` is now stored in clear text. Use `userDataBase64` if you need to provide base64-encoded user data.
- `cpuCoreCount` and `cpuThreadsPerCore` have been removed. Use the `cpuOptions` block with `coreCount` and `threadsPerCore` instead.

### `aws.eks.Cluster`

- `certificateAuthorities` has been removed. Use `certificateAuthority` instead.
- `defaultAddonsToRemoves` (a Pulumi-only patch) has been removed. Use the upstream `bootstrapSelfManagedAddons` field set to `false`, then add platform addons as `aws.eks.Addon` resources.

### `aws.elasticache.ReplicationGroup`

- `authTokenUpdateStrategy` no longer has a default value. If `authToken` is set, `authTokenUpdateStrategy` must also be explicitly set.
- The `engine` value must now be entirely lowercase. Uppercase values are no longer accepted.

### `aws.s3.Bucket`

`bucketRegion` has been added as a new output property. Use `bucketRegion` to get the bucket's region. The `region` property now refers to Enhanced Region Support.

## ECR credentials migration

The Pulumi-only `aws.ecr.getCredentials` function has been removed. Use the upstream `aws.ecr.getAuthorizationToken` instead.

<CodeGroup>

```typescript TypeScript (before v7)
const credentials = await aws.ecr.getCredentials({ registryId: registryId });
const decodedCredentials = Buffer.from(credentials.authorizationToken, "base64").toString();
const [username, password] = decodedCredentials.split(":");
```

```typescript TypeScript (v7+)
const credentials = await aws.ecr.getAuthorizationToken({ registryId: registryId });
const username = credentials.userName;
const password = pulumi.secret(credentials.password);
```

</CodeGroup>

## Deprecations

### Amazon Elastic Transcoder

Amazon Elastic Transcoder will be discontinued on **November 13, 2025**. The following resources are deprecated and will be removed in a future major release:

- `aws.elastictranscoder.Pipeline`
- `aws.elastictranscoder.Preset`

Migrate to [AWS Elemental MediaConvert](https://aws.amazon.com/blogs/media/migrating-workflows-from-amazon-elastic-transcoder-to-aws-elemental-mediaconvert/).

### CloudWatch Evidently

AWS will end support for CloudWatch Evidently on **October 17, 2025**. The following resources are deprecated and will be removed in a future major release:

- `aws.evidently.Feature`
- `aws.evidently.Launch`
- `aws.evidently.Project`
- `aws.evidently.Segment`

Migrate to [AWS AppConfig Feature Flags](https://aws.amazon.com/blogs/mt/using-aws-appconfig-feature-flags/).

### S3 global endpoint

Support for the global S3 endpoint (`s3UsEast1RegionalEndpoint = "legacy"`) is deprecated and will be removed in v8. To prepare, either remove `s3UsEast1RegionalEndpoint` from your provider configuration or set it to `"regional"` and verify your deployment.
