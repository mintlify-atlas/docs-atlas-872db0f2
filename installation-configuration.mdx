---
title: "Install and configure the Pulumi AWS Provider"
description: "Install the AWS provider SDK for your language, then configure authentication using environment variables, shared credentials, or Pulumi ESC."
---

The Pulumi AWS Provider is available as a native package in all supported Pulumi languages. Once installed, you configure it with AWS credentials using whichever authentication method suits your environment — from a local shared credentials file to short-lived OIDC tokens generated by Pulumi ESC.

## Installation

<CodeGroup>

```bash TypeScript / JavaScript
npm install @pulumi/aws
```

```bash Python
pip install pulumi-aws
```

```bash Go
go get github.com/pulumi/pulumi-aws/sdk/v7/go/aws
```

```bash .NET
dotnet add package Pulumi.Aws
```

```xml Java (Maven)
<dependency>
  <groupId>com.pulumi</groupId>
  <artifactId>aws</artifactId>
  <version>LATEST</version>
</dependency>
```

</CodeGroup>

Package registries for each language:

| Language | Package |
|---|---|
| TypeScript / JavaScript | [`@pulumi/aws`](https://www.npmjs.com/package/@pulumi/aws) on npm |
| Python | [`pulumi-aws`](https://pypi.org/project/pulumi-aws/) on PyPI |
| Go | [`github.com/pulumi/pulumi-aws/sdk/v7`](https://pkg.go.dev/github.com/pulumi/pulumi-aws/sdk/v7/go/aws) |
| .NET | [`Pulumi.Aws`](https://www.nuget.org/packages/Pulumi.Aws) on NuGet |
| Java | [`com.pulumi:aws`](https://search.maven.org/search?q=com.pulumi.aws) on Maven Central |

## Credentials setup

<Note>
Your AWS credentials are never sent to pulumi.com. Pulumi uses the AWS SDK and the credentials in your local environment to authenticate requests directly from your machine to AWS.
</Note>

Before running `pulumi up`, make sure the IAM user or role you are using has sufficient permissions to create and manage the AWS resources in your program.

<Tabs>
  <Tab title="Environment variables">
    Environment variables are useful when you need to temporarily override a credentials file, or when running in a CI/CD environment that does not have an AWS CLI installed.

    ```bash Linux / macOS
    export AWS_ACCESS_KEY_ID=<YOUR_ACCESS_KEY_ID>
    export AWS_SECRET_ACCESS_KEY=<YOUR_SECRET_ACCESS_KEY>
    export AWS_REGION=<YOUR_AWS_REGION>   # e.g. us-east-1
    ```

    ```powershell Windows
    $env:AWS_ACCESS_KEY_ID     = "<YOUR_ACCESS_KEY_ID>"
    $env:AWS_SECRET_ACCESS_KEY = "<YOUR_SECRET_ACCESS_KEY>"
    $env:AWS_REGION            = "<YOUR_AWS_REGION>"
    ```

    If you are using [temporary security credentials](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_use-resources.html), also set `AWS_SESSION_TOKEN`:

    ```bash
    export AWS_SESSION_TOKEN=<YOUR_SESSION_TOKEN>
    ```

    You can also set the region in your stack configuration file instead of an environment variable:

    ```bash
    pulumi config set aws:region us-east-1
    ```
  </Tab>

  <Tab title="Shared credentials file">
    A shared credentials file at `~/.aws/credentials` is the recommended approach for local development. It supports multiple named profiles and is read automatically by the AWS SDK.

    **Using the AWS CLI:**

    ```bash
    aws configure
    ```

    The CLI prompts for your access key, secret key, and default region, then writes `~/.aws/credentials` for you.

    **Creating the file by hand:**

    ```ini ~/.aws/credentials
    [default]
    aws_access_key_id     = <YOUR_DEFAULT_ACCESS_KEY_ID>
    aws_secret_access_key = <YOUR_DEFAULT_SECRET_ACCESS_KEY>

    [test-account]
    aws_access_key_id     = <YOUR_TEST_ACCESS_KEY_ID>
    aws_secret_access_key = <YOUR_TEST_SECRET_ACCESS_KEY>

    [prod-account]
    aws_access_key_id     = <YOUR_PROD_ACCESS_KEY_ID>
    aws_secret_access_key = <YOUR_PROD_SECRET_ACCESS_KEY>
    ```

    To select a non-default profile, use either an environment variable or Pulumi config:

    ```bash
    export AWS_PROFILE=test-account
    # or
    pulumi config set aws:profile test-account
    ```
  </Tab>

  <Tab title="EC2 instance metadata">
    When running on an EC2 instance with an attached IAM instance profile, the AWS SDK can retrieve credentials from the instance metadata service (IMDS). This is disabled by default in the Pulumi AWS provider since v3.28.1.

    To enable it, set `skipMetadataApiCheck` to `false`:

    **Using Pulumi config (default provider):**

    ```bash
    pulumi config set aws:skipMetadataApiCheck false
    ```

    **Using a named provider in code:**

    <CodeGroup>

    ```typescript TypeScript
    const provider = new aws.Provider("named-provider", {
        skipMetadataApiCheck: false,
    });
    ```

    ```python Python
    provider = pulumi_aws.Provider("named-provider", skip_metadata_api_check=False)
    ```

    ```go Go
    provider, err := aws.NewProvider(ctx, "named-provider", &aws.ProviderArgs{
        SkipMetadataApiCheck: pulumi.Bool(false),
    })
    ```

    ```csharp .NET
    var provider = new Aws.Provider("named-provider", new Aws.ProviderArgs
    {
        SkipMetadataApiCheck = false,
    });
    ```

    </CodeGroup>
  </Tab>

  <Tab title="OIDC / WebIdentity">
    OIDC lets you run Pulumi on services like GitHub Actions, GitLab CI, or Azure DevOps and access AWS without storing long-lived credentials. AWS issues temporary credentials in exchange for a signed OIDC ID token from the CI provider.

    **On GitHub Actions**, use the official `configure-aws-credentials` action:

    ```yaml .github/workflows/deploy.yml
    - uses: aws-actions/configure-aws-credentials@v4
      with:
        aws-region: ${{ env.AWS_REGION }}
        role-session-name: pulumi-deploy
        role-to-assume: arn:aws:iam::<ACCOUNT_ID>:role/<ROLE_NAME>
    ```

    **On other platforms**, configure `assumeRoleWithWebIdentity` in your stack config:

    ```yaml Pulumi.<stack>.yaml
    config:
      aws:assumeRoleWithWebIdentity:
        roleArn: arn:aws:iam::<ACCOUNT_ID>:role/<ROLE_NAME>
        webIdentityTokenFile: /path/to/token.txt
    ```

    Refer to the AWS docs on [web identity federation](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_providers_oidc.html) for how to configure the trust relationship between your CI provider and AWS.
  </Tab>

  <Tab title="Pulumi ESC">
    Pulumi ESC (Environments, Secrets, and Configuration) can generate short-lived AWS credentials dynamically via OIDC, removing the need to manage static access keys anywhere.

    **Step 1 — Configure OIDC between Pulumi and AWS.** Follow the [Configuring OpenID Connect for AWS guide](https://www.pulumi.com/docs/pulumi-cloud/oidc/aws/). Your ESC environment file will look similar to this:

    ```yaml ESC environment
    values:
      aws:
        login:
          fn::open::aws-login:
            oidc:
              duration: 1h
              roleArn: <your-oidc-iam-role-arn>
              sessionName: pulumi-environments-session
      environmentVariables:
        AWS_ACCESS_KEY_ID:     ${aws.login.accessKeyId}
        AWS_SECRET_ACCESS_KEY: ${aws.login.secretAccessKey}
        AWS_SESSION_TOKEN:     ${aws.login.sessionToken}
        AWS_REGION:            us-east-1
    ```

    **Step 2 — (Optional) Move Pulumi config into ESC.** Add a `pulumiConfig` block to expose configuration values to Pulumi IaC without duplicating them in `Pulumi.<stack>.yaml`:

    ```yaml ESC environment
    values:
      aws:
        login:
          fn::open::aws-login:
            oidc:
              duration: 1h
              roleArn: <your-oidc-iam-role-arn>
              sessionName: pulumi-environments-session
      pulumiConfig:
        aws:region: us-east-1
        aws:accessKey:  ${aws.login.accessKeyId}
        aws:secretKey:  ${aws.login.secretAccessKey}
        aws:token:      ${aws.login.sessionToken}
        aws:assumeRoles:
          - roleArn: arn:aws:iam::058111598222:role/OrganizationAccountAccessRole
    ```

    **Step 3 — Import the environment into your stack.** Update `Pulumi.<stack>.yaml`:

    ```yaml Pulumi.<stack>.yaml
    environment:
      - <your-environment-name>
    ```

    Run `pulumi preview` to verify that credentials are resolved correctly. Make sure no local AWS credentials are set, as they would take precedence over the ESC-generated ones.
  </Tab>
</Tabs>

## Configuration options

Use `pulumi config set aws:<option>` or pass options to the constructor of `new aws.Provider`. The table below covers the most commonly used options.

| Option | Required | Description |
|---|---|---|
| `region` | Yes | AWS region for all operations. Examples: `us-east-1`, `eu-west-1`, `ap-southeast-2`. |
| `accessKey` | No | AWS access key ID. Equivalent to `AWS_ACCESS_KEY_ID`. |
| `secretKey` | No | AWS secret access key. Equivalent to `AWS_SECRET_ACCESS_KEY`. |
| `token` | No | Session token for temporary credentials. Equivalent to `AWS_SESSION_TOKEN`. |
| `profile` | No | Named profile from `~/.aws/credentials`. Defaults to the `default` profile. |
| `sharedCredentialsFile` | No | Path to a credentials file. Defaults to `~/.aws/credentials`. |
| `assumeRoles` | No | List of IAM roles to assume. See nested properties below. |
| `assumeRoleWithWebIdentity` | No | IAM role to assume via OIDC/WebIdentity. See nested properties below. |
| `defaultTags` | No | Tags applied to every resource managed by this provider. Accepts a `tags` map. |
| `ignoreTags` | No | Tags to ignore across all resources. Accepts `keys` and `keyPrefixes` lists. |
| `allowedAccountIds` | No | Allowlist of AWS account IDs. Prevents accidental deployments to the wrong account. Conflicts with `forbiddenAccountIds`. |
| `forbiddenAccountIds` | No | Blocklist of AWS account IDs. Conflicts with `allowedAccountIds`. |
| `skipMetadataApiCheck` | No | Disable the EC2 instance metadata endpoint check. Defaults to `true`. Set to `false` to enable IMDS-based authentication. |
| `skipCredentialsValidation` | No | Skip STS credential validation. Useful for non-standard AWS-compatible APIs. |
| `skipRegionValidation` | No | Skip static region name validation. Useful for private or non-public regions. |
| `skipRequestingAccountId` | No | Skip IAM/STS account ID lookup. |
| `maxRetries` | No | Maximum number of AWS API request retries before returning an error. |
| `retryMode` | No | Retry strategy: `standard` or `adaptive`. Can also be set via `AWS_RETRY_MODE`. |
| `insecure` | No | Allow insecure (non-TLS) requests. Defaults to `false`. |
| `s3ForcePathStyle` | No | Force path-style S3 addressing (`s3.amazonaws.com/BUCKET/KEY`) instead of virtual-hosted style. |
| `dynamodbEndpoint` | No | Override the DynamoDB endpoint URL. Useful for connecting to `dynamodb-local`. |
| `kinesisEndpoint` | No | Override the Kinesis endpoint URL. Useful for connecting to `kinesalite`. |

### assumeRoles nested properties

| Property | Description |
|---|---|
| `roleArn` | ARN of the IAM role to assume. |
| `sessionName` | Session name used when assuming the role. |
| `externalId` | External ID required by the role trust policy. |
| `durationSeconds` | Maximum session duration in seconds. |
| `policy` | Inline IAM policy JSON to further restrict the session. |
| `policyArns` | Set of managed policy ARNs to further restrict the session. |
| `tags` | Session tags to attach. |
| `transitiveTagKeys` | Tag keys to pass through to subsequent sessions. |

### assumeRoleWithWebIdentity nested properties

| Property | Description |
|---|---|
| `roleArn` | ARN of the IAM role to assume. |
| `sessionName` | Session name used when assuming the role. |
| `webIdentityToken` | Inline OIDC ID token value. |
| `webIdentityTokenFile` | Path to a file containing the OIDC ID token. |
| `durationSeconds` | Maximum session duration in seconds. |
| `policy` | Inline IAM policy JSON to further restrict the session. |
| `policyArns` | Set of managed policy ARNs to further restrict the session. |

### Setting nested configuration

Use `--path` to set nested properties from the CLI:

```bash
pulumi config set --path aws:assumeRoles[0].roleArn arn:aws:iam::058111598222:role/OrganizationAccountAccessRole
pulumi config set --path aws:defaultTags.tags.Environment production
```
